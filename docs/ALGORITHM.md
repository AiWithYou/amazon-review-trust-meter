# 判定アルゴリズム v2.0.4

## 1. 目的

このアルゴリズムは、Amazon.co.jpの商品ページから取得できる公開情報だけを使い、レビュー操作を断定するのではなく、**追加確認が必要な複合兆候**を検出します。

最優先する設計目標は、検出率を無理に上げることではなく、正常商品を要注意扱いする誤警告を減らすことです。そのため、星5比率、短文、投稿日集中、購入確認なし、無名ブランドといった単独特徴では強い判定を出しません。一方、商品名と説明の間で複数の具体的な仕様値が矛盾する場合は、レビュー操作とは別の購入前確認材料として扱います。

## 2. 入力

```ts
interface ProductInput {
  averageRating: number | null;
  reviewCount: number | null;
  distribution: Record<1 | 2 | 3 | 4 | 5, number | null>;
  title: string;
  brand: string;
  details: string;
  reviews: Array<{
    id?: string;
    reviewerId?: string;
    stars: number | null;
    title?: string;
    body: string;
    date: string;
    verified: boolean | null;
    vine: boolean;
    variation?: string;
    helpfulVotes?: number | null;
    imageCount?: number;
  }>;
}
```

分析対象レビューは最大24件、現在のDOM抽出は最大20件です。

## 3. 前処理

本文には次の正規化を行います。

1. Unicode NFKC正規化
2. 小文字化
3. URL、メール、長い注文番号らしき文字列の除去
4. 数値を共通記号へ置換
5. 空白、句読点、記号の除去

数値を共通化する理由は、テンプレート文の数値だけを差し替えたレビューを検出するためです。ただし、具体性判定には正規化前の本文を使うため、数値を含む実体験レビューが一律に汎用文扱いされることはありません。

## 4. 本文の汎用度

本文長を基礎に、次の特徴で0〜1へ変換します。

- 「良い商品です」などの完全一致定型文
- 数量、時間、寸法、接続、充電、故障、返品などの具体語
- 「ただし」「一方」などの長所・短所を併記する表現
- 複数文と文字の多様性

`genericness >= 0.68` を汎用本文として扱います。短いこと自体は弱い特徴であり、高評価または低評価側へ多数集中した場合だけ加点します。

## 5. 本文類似クラスタ

日本語の形態素解析や外部モデルに依存せず、文字2-gramまたは3-gramを使います。

2本文 `A, B` の類似度は概ね次です。

```text
similarity =
  (0.58 × Jaccard(A, B) + 0.42 × Containment(A, B))
  × sqrt(minLength / maxLength)
```

片方がもう片方を包含する場合は包含率も考慮します。本文長に応じた閾値を超えたペアを辺としてUnion-Findで連結し、クラスタを作ります。

クラスタ強度には次を使います。

- 最大クラスタ件数
- 分析可能本文に占める比率
- クラスタ内の平均類似度
- クラスタ内の辺密度

2件だけの類似は、ほぼ完全一致でなければ強く加点しません。連鎖的に結ばれた疎なクラスタは、辺密度によって弱くなります。

## 6. 投稿日バースト

日付を取得できたレビューが6件以上ある場合、14日、30日、90日の各窓で最大集中区間を探します。

取得レビュー全体の期間を `spanDays`、窓幅を `windowDays` とすると、均一な投稿を仮定した期待比率を次で近似します。

```text
expectedRatio = windowDays / spanDays
expectedCount = n × expectedRatio
z = (observedCount - expectedCount)
    / sqrt(n × expectedRatio × (1 - expectedRatio))
```

実際の強度は、期待比率との差、z値、日付取得件数から0〜1へ変換します。観測期間自体が窓幅と近い場合は判定しません。

投稿日集中だけでは強い判定にしません。次のいずれかが重なる必要があります。

- 高評価または低評価へ方向が揃う
- 同じ集中区間に類似本文クラスタが含まれる
- 汎用本文が集中する
- 星別分布の偏りと一致する

強い本文重複や未購入レビューがなくても、★5が75%以上、★1が8%以上、中間評価が18%以下という中程度の二極化があり、短期窓に高評価が85%以上の方向で集中する場合は、両者の重なりを弱い複合兆候として加点します。日付集中または二極化の片方だけでは「確認推奨」にしません。

Vineが集中区間の半数以上を占める場合は、発売時の正規レビュー集中候補として補足表示に回し、バースト加点を抑止します。

## 7. レビュー単位の疑わしさ

補正評価用に、各レビューへ0〜0.9の内部リスクを付けます。

主な要素は次のとおりです。

- 3件以上の類似本文クラスタへの所属
- 極端な汎用本文
- 購入確認なし。ただしVineを除く
- 評価方向が揃った不自然な投稿集中への所属
- 強い宣伝定型句

具体性の高い長文は小さく減算し、Vineレビューはリスクを大きく縮小します。この内部値はUIに「不正確率」として表示しません。

## 8. 特徴群と上限

同じ原因から派生した特徴を何重にも加算しないよう、特徴を群に分けて上限を設けます。

| 群 | 上限 |
|---|---:|
| 商品情報 | 65 |
| 評価分布 | 16 |
| レビュー本文 | 28 |
| 投稿時系列 | 14 |
| 投稿属性 | 10 |
| 複合兆候 | 22 |
| レビュー適用範囲 | 6 |

商品名の長さ、語の反復、宣伝表現と星分布は単独では弱い根拠です。レビュー兆候には、独立した特徴群の重なりとレビュー件数に応じた追加上限をかけます。商品記載はレビュー兆候と別に集計し、連続時間、防水・防塵等級、電池容量、発光パターン数の複数項目が矛盾する場合は、レビュー兆候が少なくても購入前確認度へ反映します。

また、「初・唯一」「最上級・革新」「数値で示した大幅な性能向上」「独自・特許」という4種類すべての強い訴求が商品名・説明内に重なる場合は、事実認定をせず、検証が必要な商品記載として弱く加点します。画像添付は表示標本の偏りを受けやすいため補足表示だけに留めます。

注意度には次の内訳があります。

- `reviewRiskScore`：星分布、本文、時系列、投稿属性、複合兆候、バリエーション混在
- `listingRiskScore`：商品記載の長さ、反復、宣伝表現、具体的な仕様矛盾

ブランド名は加点にも減点にも使いません。括弧などで日本語名と英字名が併記されている場合は、いずれかの別名が商品名に含まれていれば表記一致とします。

## 9. 主な複合兆候

### 高評価操作候補

- 類似本文クラスタ
- 高評価側への投稿日集中
- 購入確認のない高評価が多数
- ★5偏重
- ★5と★1の中程度の二極化と、短期高評価集中の重なり
- 高評価中心の分布と、参考票が突出した低評価の重なり

### 低評価攻撃候補

- 類似本文クラスタ
- 低評価側への投稿日集中
- 購入確認のない低評価が多数
- ★1比率の増加

高評価だけでなく、競合商品を下げるレビュー攻撃も同じ原理で扱います。

## 10. 分析材料

UIで表示する「分析材料」は危険度、判定精度、不正確率ではなく、分析に使えるデータの充足度です。コード上のプロパティ名は `confidence` を維持します。

```text
confidence =
  0.15 × 平均評価・件数
+ 0.25 × 星別分布
+ 0.10 × 商品情報
+ 0.25 × 本文取得量
+ 0.15 × 日付取得量
+ 0.10 × 投稿属性取得量
```

表示レビューしか取得しないため、最大値は86%です。レビューがない、少ない、星別割合の合計が丸め誤差を超えて崩れている、といった場合はさらに下げます。

## 11. 補正評価

補正評価は注意度を★へ線形変換しません。星別分布を基準に、表示レビューから推定した各星の内部リスクで小さく再重み付けします。折りたたみ表示の★はこの補正評価であり、商品記載の注意度とは別の値です。

```text
adjustedWeight[star] =
  originalWeight[star] × (1 - correctionStrength × bucketRisk[star])
```

補正強度は次で抑えます。

- 分析材料
- 表示レビュー件数
- 表示レビューの星の網羅性
- 表示レビュー平均とAmazon平均の差

表示レビューは無作為標本ではないため、最大補正幅は±0.7に制限し、通常はさらに小さくなります。これはAmazon平均を置き換える値ではありません。

## 12. 現在の限界

ローカル版の最大の制約は、商品横断のレビュー・レビュアーグラフを持たないことです。本格的な組織検出には、少なくとも次の公開情報を複数商品から継続的に集める必要があります。

- reviewer ID
- ASIN
- 投稿日
- 星評価
- 本文フィンガープリント
- Vine、購入確認、バリエーション

そのうえで、同じレビュアー群が短期間に同じ複数商品へ投稿する共同行動を検出します。ただし、外部送信を伴う場合は、現在の「完全ローカル・追加権限なし」というプライバシー方針と別機能として設計すべきです。

## 13. 校正方針

現在の係数は説明可能なルールベースの初期値です。将来、十分な検証データを作る場合は次を守ります。

- レビュー単位のランダム分割をしない
- ASIN、ブランド、期間をまたいで学習・評価を分離する
- 正常な新発売、セール、Vine、バリエーション統合を負例へ含める
- 高評価操作と低評価攻撃を別々に評価する
- accuracyより、要注意表示の適合率、誤警告率、PR-AUC、確率校正を優先する
- 係数変更時は、固定回帰テストと実ページの匿名化フィクスチャを併用する
